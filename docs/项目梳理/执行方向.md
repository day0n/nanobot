# OpenCreator Agent 执行方向

## 1. 目标定义

### 1.1 版本目标

本期目标以 `2026年4月20日` 为硬截止，产品口径直接对标“切片3”，但研发落地要按切片3的能力依赖倒推实施。

本期建议的**可交付口径**定义为：

> 用户给出目标或素材后，Agent 能结合用户意图搜索并筛选参考爆款内容，分析视频/参考内容结构，自动生成可编辑的 OpenCreator workflow，并可直接落到当前画布，必要时触发运行。

### 1.2 本期不建议承诺的口径

以下能力不要作为 `4月20日` 的硬承诺：

- 全自动“直接交付成片”
- 所有平台的爆款视频无障碍抓取
- 完整自动化剪辑合成闭环
- 多轮自我评估和自动 retry 到最优

这些能力不是做不了，而是不适合在当前时间窗内与现有代码基础一起并发推进。

### 1.3 本期核心策略

对外目标对标切片3，对内研发拆成 4 层能力：

1. 用户意图理解
2. 参考内容搜索与筛选
3. 视频结构拆解与 workflow 映射
4. workflow 方案生成、写入画布、执行反馈

切片3本质上是切片1和切片2的组合增强版，因此内部研发不能跳过切片1和切片2，只能把它们作为切片3的前置能力实现。

## 2. 总体技术路线

### 2.1 继续复用当前项目的主干

当前项目已经具备这些可复用基础：

- `FastAPI + SSE` 对话入口
- `AgentLoop` 主编排循环
- `ToolRegistry`
- `SessionManager`
- `Mem0Memory`
- `WorkflowEngine`
- `OpenCreator workflow tools`
- `Mongo + Redis + RabbitMQ`

因此本期不建议推倒重来，而建议沿现有主干演化。

### 2.2 不预设固定 Agent 数量

本期不建议先把系统绑死成“1 个主 Agent + 2 个子 Agent”。

更合理的原则是：

> 先按职责拆出逻辑角色，再决定哪些角色本期需要独立成运行时 Agent，哪些更适合做 Tool / Service。

原因：

- 真正该拆成 Agent 的，是需要独立判断、独立策略、独立上下文的环节。
- 更偏确定性的环节，例如视频下载、抽帧、转录、知识检索、workflow 状态监听，更适合做 Tool / Service。
- 当前 deadline 很紧，过早做太多平权 Agent，会增加协调和调试成本。

### 2.3 建议的逻辑角色拆分

从切片3完整能力看，建议先按 5 个逻辑角色设计：

1. `Main Orchestrator`
2. `Trend Scout Agent`
3. `Video Analyst Agent`
4. `Workflow Librarian`
5. `Workflow Builder Agent`

其中：

- `Trend Scout Agent` 负责参考爆款搜索与筛选
- `Video Analyst Agent` 负责视频拆解与结构化分析
- `Workflow Librarian` 负责从 `200+` 商业化 workflow 与最佳实践中做 pattern 检索
- `Workflow Builder Agent` 负责把意图、参考内容、分析结果、pattern 结果整合成 workflow plan，并与画布交互

### 2.4 本期运行时建议：1 主 + 3 专职 + 若干工具服务

虽然逻辑上建议拆成 5 个角色，但本期运行时不建议真的落成 5 个独立 Agent。

本期更稳的落地方式是：

1. `Main Orchestrator`
2. `Trend Scout Agent`
3. `Video Analyst Agent`
4. `Workflow Builder Agent`

并把以下能力先做成 Tool / Service，而不是独立 Agent：

- `Workflow Librarian`：先做成 `retrieve_workflow_patterns` Tool + 知识检索层
- `video_ingest`
- `extract_keyframes`
- `transcribe_video`
- `knowledge_search`
- `workflow_run_monitor`

这样做的好处是：

- 角色边界足够清晰
- 系统协调成本不会爆炸
- 能在 `4月20日` 前保住交付节奏

### 2.5 本期运行时 Agent 的职责边界

| Agent / Role | 类型 | 职责 | 不负责 |
| --- | --- | --- | --- |
| `Main Orchestrator` | Runtime Agent | 理解意图、路由任务、汇总结果、维护 session 状态、最终回复用户 | 深度搜索、深度视频分析、直接持有全部 workflow 工具 |
| `Trend Scout Agent` | Runtime Agent | 搜索参考爆款、筛选候选内容、输出 `reference_pack` | 直接编辑 workflow、直接运行 workflow |
| `Video Analyst Agent` | Runtime Agent | 拆解视频结构、重建脚本、输出 `video_analysis_report` | 管理全局会话、直接编辑 workflow |
| `Workflow Builder Agent` | Runtime Agent | 匹配 workflow pattern、生成 `workflow_plan`、调用 workflow tools 落画布或运行 | 泛化 web 搜索、管理全局会话 |
| `Workflow Librarian` | Tool / Service | 从 `200+` 商业化 workflow 和最佳实践中检索 pattern | 不直接回复用户、不直接改画布 |

### 2.6 本期的关键架构原则

1. 统一运行时，不统一上下文  
底层都可以复用 `AgentLoop`，但每个 agent 的 `tools / skills / memory scope / prompt` 必须不同。

2. 主 Agent 不加载全部记忆与全部工具  
主 Agent 只持有最小必要能力，避免强中心化和 prompt 污染。

3. 所有中间结果必须结构化  
本期最重要的不是“更多自然语言回复”，而是形成结构化的 `reference_pack / video_analysis / workflow_plan`。

4. 所有高价值中间结果都进入数据闭环  
这是把“200+成熟商业化工作流”变成长期壁垒的关键。

5. 不是所有能力都要做成 Agent  
有判断的能力优先做 Agent，确定性的能力优先做 Tool / Service。

### 3.2 `4月20日` 的验收标准

到 `2026年4月20日`，至少要做到：

1. 用户只输入需求文案，可以生成 workflow plan，并能落到画布。
2. 用户上传视频或提供可分析的视频链接，可以拆出视频结构并生成 workflow plan。
3. 用户表达目标后，系统能搜索并筛选参考内容，输出参考包，再据此生成 workflow plan。
4. workflow plan 不是纯文本，而是可映射到 `nodes/edges` 的结构化产物。
5. 全链路核心中间结果都有数据落盘和可追踪状态。

## 4. 数据闭环总览

### 4.1 统一闭环实体

本期建议把所有核心中间产物统一成 7 个实体：

1. `task_run`
2. `reference_asset`
3. `reference_pack`
4. `video_analysis_report`
5. `pattern_match_result`
6. `workflow_plan`
7. `execution_result`

### 4.2 闭环路径

```text
User Query
  -> task_run
  -> reference search / ingest
  -> reference_pack
  -> video_analysis_report
  -> pattern_match_result
  -> workflow_plan
  -> edit_workflow / run_workflow
  -> execution_result
  -> user feedback / acceptance
  -> 回写知识库与 pattern library
```

### 4.3 闭环原则

- 会话态数据与业务资产分离
- Redis 只做热状态和短期编排，不做最终真相
- Mongo 做主存储
- 向量库只存“可复用知识”，不存所有瞬时会话内容
- 用户接受过、成功率高的 workflow 和视频拆解结果要反哺知识库

## 7. 向量化存储设计

### 7.1 向量库建议

本期优先建议：

- 继续使用 `MongoDB Atlas Vector Search`

原因：

1. 当前项目已经使用 Mongo
2. `mem0` 已经走 Mongo 向量存储思路
3. 截止时间紧，不建议为了本期再引入新的核心基础设施

不建议本期新增：

- 独立 Qdrant
- 独立 Milvus
- 独立 ES + 向量双栈

这些都能做，但对 `4月20日` 的 deadline 不划算。

### 7.2 哪些数据进入向量化存储

只把“可复用知识”放进向量库，不把所有任务中间态都丢进去。

建议进入向量化存储的内容：

1. `200+成熟商业化 workflow` 的模式描述
2. `docs/最佳实战` 的业务知识文档
3. 节点/模型说明文档
4. 已人工确认质量较高的 `video_analysis_report`
5. 已人工确认质量较高的 `workflow_plan`
6. 产品 FAQ、产研结论、复盘文档

不建议进入向量化存储的内容：

- 所有 session messages
- 所有 tool outputs
- 所有临时用户视频
- 所有失败或未确认的分析结果

### 7.3 向量集合建议：`knowledge_chunks`

建议结构：

```json
{
  "_id": "kchunk_01J...",
  "document_id": "kdoc_01J...",
  "chunk_index": 0,
  "text": "UGC口播 workflow 的共享语义层应该先做卖点分析...",
  "embedding": [0.0123, -0.0831, 0.4412],
  "metadata": {
    "source_type": "workflow_pattern",
    "source_ref_id": "pattern_ugc_lipsync_v2",
    "tags": ["ugc", "lipsync", "beauty"],
    "scenario": "ugc_ad",
    "input_signature": ["product_image", "product_url"],
    "output_signature": ["marketing_video"],
    "curated": true
  },
  "created_at": "2026-03-27T00:00:00+08:00"
}
```

### 7.4 检索策略建议

本期知识检索建议采用“混合检索 + 轻 rerank”：

1. 元数据过滤  
按 `scenario / input_signature / output_signature / tags`

2. 向量召回  
按语义近似检索 topK

3. 轻量 rerank  
用规则分数或轻量 LLM 对 topK 做重排

建议不要一上来做复杂 RAG pipeline，先做：

- metadata filter
- vector topK
- heuristic rerank

足够本期使用。

## 8. 是否需要引入其他工具

结论：需要，但必须控制在**最小增量**。

### 8.1 建议新增的外部依赖/服务

#### 1. `ffmpeg / ffprobe`

用途：

- 读取视频基础元数据
- 抽关键帧
- 生成封面

这是视频分析链路的最低成本依赖，必须引入。

#### 2. 对象存储/文件服务

用途：

- 存用户上传视频
- 存下载后的参考视频
- 存抽帧图片

如果团队已有 OSS/S3/内部文件服务，直接复用，不要自己做文件系统方案。

#### 3. 视频转录能力

优先建议：

- 先用现有大模型多模态能力或已有 ASR 服务

不建议本期自己再单独上复杂语音服务平台，只要能稳定产出 transcript 即可。

#### 4. 平台视频采集 Adapter

如果切片3必须支持“搜索真实爆款并分析内容”，就需要一个**后端采集 adapter**，而不是让 agent 直接 shell 抓视频。

建议做法：

- Agent 调内部 `video_ingest` 能力
- 由后端 adapter 去处理不同平台链接

不建议：

- 把 `yt-dlp` 或各类抓取脚本直接暴露给 agent tool

原因：

- 合规风险
- 平台限制
- tool 层不稳定

### 8.2 本期不建议引入的新核心基础设施

- 新的向量数据库
- 新的消息队列
- 新的主数据库
- 大而全的编排引擎

现有 `Mongo + Redis + RabbitMQ + FastAPI` 足够支撑本期。

## 11. 知识库建设方案

### 11.1 知识源组成

本期知识库建议由 5 类来源组成：

1. `200+成熟商业化 workflow`
2. `docs/最佳实战`
3. 节点/模型/规格文档
4. 已确认质量高的视频拆解报告
5. 已确认质量高的 workflow plan 和复盘文档

### 11.2 最重要的知识源：200+商业化 workflow

这些 workflow 不是只拿来“当文件存档”，而是要做成两层资产：

#### 第一层：结构化资产

放进 `workflow_pattern_library`

包括：

- 适用场景
- 输入输出签名
- 节点类型签名
- DAG 结构特征
- 复用模板
- 质量标签

#### 第二层：语义知识资产

放进 `knowledge_documents + knowledge_chunks`

包括：

- workflow 的自然语言摘要
- 关键设计原因
- 为什么这样分支
- 为什么这样融合
- 适合什么产品、什么内容、什么输入条件

### 11.3 知识入库流程

建议离线批处理 pipeline：

1. 导出 workflow 原始 `nodes/edges`
2. 计算 graph signature
3. 用 LLM 生成 semantic summary
4. 生成 `knowledge_document`
5. 切 chunk
6. 生成 embedding
7. 写入 `knowledge_chunks`

### 11.4 知识库技术选型

本期建议：

- 存储：MongoDB
- 向量：MongoDB Atlas Vector Search
- 切片：离线 Python 脚本
- embedding：`text-embedding-3-small`
- 检索：metadata filter + vector topK + heuristic rerank

### 11.5 如果内部已有 Linglong

如果团队已经在用 Linglong 做内部知识切片，建议这样分工：

- Linglong 负责上游知识整理、清洗、切片产出
- 当前 Agent 项目负责在线检索、召回和工具封装

不要把 Linglong 直接当在线 agent runtime，用它做内容生产层更合适。

### 11.6 知识库如何调用

本期建议不要让 agent 直接“读所有知识库全文”，而是：

1. agent 调 `knowledge_search` 或 `retrieve_workflow_patterns`
2. tool 返回 topK 片段和 pattern 摘要
3. agent 基于返回内容做分析/规划

这比把整个知识库塞进 prompt 成本更低、稳定性更高。

## 13. 关键建议

### 13.1 不要把全部记忆塞给主 Agent

建议：

- `Main Orchestrator` 只拿 session summary + user preference
- `Trend Scout Agent` 只拿搜索策略和参考筛选知识
- `Video Analyst Agent` 只拿视频拆解 rubric
- `Workflow Builder Agent` 只拿 pattern KB、节点规则、workflow 规划规则

### 13.2 不要把 workflow 知识放进用户长期记忆

`Mem0Memory` 继续用于用户偏好即可，不要把：

- workflow pattern
- 视频拆解模板
- node 规则

放进 mem0。那是领域知识，不是用户记忆。

### 13.3 不要让 agent 直接抓平台视频

建议：

- agent 调后端 `ingest_video_asset`
- 后端 adapter 做平台适配

这样可以把：

- 合规
- 稳定性
- 重试

从 agent prompt 层剥离出去。

### 13.4 `Workflow Librarian` 本期先做成 Tool / Service

建议：

- 逻辑上保留 `Workflow Librarian` 这个角色
- 但本期不要急着把它做成独立运行时 Agent
- 先通过 `retrieve_workflow_patterns + knowledge_search + pattern_match_result` 承担职责

这样可以既保留未来扩展空间，又不增加本期协调复杂度。

### 13.5 200+ workflow 是本期最重要的护城河

本期最有价值的不是“模型多厉害”，而是：

- 你们已经有 200+ 成熟商业化 workflow
- 这批资产可以直接变成 workflow pattern library

如果这层建好，Agent 的 workflow 规划质量会明显高于只靠大模型现编。

### 13.6 本期优先做“可用”和“可解释”
