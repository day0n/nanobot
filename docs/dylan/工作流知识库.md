# OpenCreator Node/Edge 知识库

本文档是写给“在后端 Python 项目里操作 OpenCreator 工作流数据的 agent”的。

目标不是解释前端组件实现，而是给 agent 一个稳定的工作流契约：

1. 哪些信息必须理解，才能正确创建/修改工作流
2. 哪些信息只需要保留，不需要推理
3. 哪些信息应该忽略或清理
4. 每种节点接受什么输入、输出什么、需要什么参数、什么条件下可运行
5. 新增/修改/删除节点与连线时，数据结构应该如何变化

---

## 一、Agent 到底需要知道什么

### 1.1 必须理解

- `nodes[]` 和 `edges[]` 的结构
- `node.type` 才是节点类型的真正来源
- 每种节点的输入类型、输出类型、必填输入、最大连线数
- 每种节点真正影响运行的 `data` 字段
- 哪些节点可执行，哪些节点不可执行
- 建立连线前必须做哪些校验
- 一个节点在什么条件下允许运行

### 1.2 只需保留，通常不用推理

这些字段对“工作流是否能跑”通常不是核心，但如果你在编辑已有工作流，应该尽量保留：

- `position`
- `selected`
- `data.label`
- `data.description`
- `data.themeColor`
- `data.inputHint`
- `data.modelCardColor`
- `data.hideResults`
- `data.userStatus`
- `stickyNodesNode` 的视觉字段
- `groupNode` 的展示字段

### 1.3 应忽略或主动清理

这些字段不应该被 backend agent 当成真实业务语义：

- `data.estimatedTime`
- `data.isNodeConnected`
- `data.isTextPinConnected`
- `data.isImagePinConnected`
- `data.local_file`

另外：

- 不要依赖 `data.type`
- 当前真实的节点类型来源是 `node.type`
- `data.type` 在类型定义里存在，但创建节点时并不会稳定写入

## 二、工作流的最小数据契约

一个工作流核心上就是：

```json
{
  "nodes": [],
  "edges": []
}
```

### 2.1 Node 外层结构

每个节点至少满足：

```json
{
  "id": "textGenerator-1234567890-abcd12",
  "type": "textGenerator",
  "position": { "x": 100, "y": 200 },
  "selected": false,
  "data": {}
}
```

字段说明：

| 字段         | 类型      | 是否必须 | 说明                                                                 |
| ------------ | --------- | -------: | -------------------------------------------------------------------- |
| `id`         | `string`  |       是 | 节点唯一 ID。当前前端默认格式为 `{nodeType}-{timestamp}-{nanoid(6)}` |
| `type`       | `string`  |       是 | 节点类型。**这是唯一可信的节点类型来源**                             |
| `position.x` | `number`  |       是 | UI 布局用                                                            |
| `position.y` | `number`  |       是 | UI 布局用                                                            |
| `selected`   | `boolean` |       否 | UI 选中态                                                            |
| `data`       | `object`  |       是 | 节点参数                                                             |

### 2.2 Edge 结构

每条边至少满足：

```json
{
  "id": "customEdge-sourceId-targetId-text-text",
  "type": "customEdge",
  "source": "textInput-1",
  "target": "textGenerator-2",
  "sourceHandle": "text",
  "targetHandle": "text"
}
```

字段说明：

| 字段           | 类型           | 是否必须 | 说明                                                                         |
| -------------- | -------------- | -------: | ---------------------------------------------------------------------------- |
| `id`           | `string`       |       是 | 边唯一 ID                                                                    |
| `type`         | `'customEdge'` |       是 | 当前只使用这一种边类型                                                       |
| `source`       | `string`       |       是 | 源节点 ID                                                                    |
| `target`       | `string`       |       是 | 目标节点 ID                                                                  |
| `sourceHandle` | `string`       | 建议必填 | 源 Pin 类型，如 `text` / `image` / `video` / `audio`                         |
| `targetHandle` | `string`       | 建议必填 | 目标 Pin 类型，如 `text` / `image` / `video` / `audio` / `subject` / `style` |
| `selected`     | `boolean`      |       否 | UI 选中态                                                                    |

## 四、Pin 和连线类型规则

### 4.1 Pin 类型集合

当前系统里真实使用的 Pin 类型只有 6 个：

- `text`
- `image`
- `video`
- `audio`
- `subject`
- `style`

### 4.2 类型兼容规则

只有以下连接是允许的：

| 源输出  | 可连接到的目标输入          |
| ------- | --------------------------- |
| `text`  | `text`                      |
| `image` | `image`、`subject`、`style` |
| `video` | `video`                     |
| `audio` | `audio`                     |

补充说明：

- `subject` 和 `style` 是 `image` 的别名
- 也就是说，`image` 输出可以接到 `subject` 或 `style`
- 反过来不要把 `subject` 当成一个独立输出类型

### 4.3 建边前必须做的 4 个校验

任意新连线都必须同时满足：

1. 不能自连：`source !== target`
2. `sourceHandle` 与 `targetHandle` 必须类型兼容
3. 目标 Pin 未超过最大连线数
4. 加这条边之后不能形成环

如果任一条件不满足，这条边都不应该创建。

## 五、节点公共字段

## 5.1 大多数节点都有的基础字段

除静态节点外，通常都带这些字段：

| 字段          | 类型     | 说明             |
| ------------- | -------- | ---------------- |
| `label`       | `string` | 节点名           |
| `description` | `string` | 节点描述         |
| `themeColor`  | `string` | 展示颜色         |
| `inputHint`   | `string` | 用户可编辑标题   |
| `status`      | `string` | 节点状态         |
| `workflowId`  | `string` | 所属 workflow ID |

`status` 允许值：

- `idle`
- `ready`
- `running`
- `success`
- `pause`
- `select`
- `failed`
- `stopped`

### 5.2 所有任务节点共有字段

所有任务节点都会有：

| 字段             | 类型       | 说明                 |
| ---------------- | ---------- | -------------------- |
| `selectedModels` | `string[]` | 已选模型 ID          |
| `isSelectMode`   | `boolean`  | UI 选择模型状态      |
| `modelConfigs`   | `object`   | 按模型 ID 索引的配置 |
| `modelCardColor` | `string`   | UI 展示字段          |
| `hideResults`    | `boolean`  | UI 展示字段          |
| `userStatus`     | `string`   | UI/状态字段          |

### 5.3 `modelConfigs` 的含义

`modelConfigs` 是：

```json
{
  "<modelId>": { "...": "..." }
}
```

它是“按模型 ID 存配置”的容器。

不要把它理解成统一 schema。不同节点会往里面放不同参数。

### 5.4 `model_options` 的含义

`model_options` 是“节点级别的配置对象”，不是按模型 ID 分桶。

它的 schema 依节点类型而变。

## 6.1 输入节点

输入节点不执行，只提供上游原始数据。

### `textInput`

- 输入：无
- 输出：`text`
- 最大输入连线：无
- 关键字段：

| 字段        | 类型     | 默认值 | 说明               |
| ----------- | -------- | ------ | ------------------ |
| `inputText` | `string` | `''`   | 富文本 HTML 字符串 |

- 上游满足条件：`inputText` 去除富文本标签后仍有非空文本

### `imageInput`

- 输入：无
- 输出：`image`
- 关键字段：

| 字段          | 类型     | 默认值 | 说明               |
| ------------- | -------- | ------ | ------------------ |
| `imageBase64` | `string` | `''`   | 图片 URL 或 base64 |

- 上游满足条件：`imageBase64` 非空

### `audioInput`

- 输入：无
- 输出：`audio`
- 关键字段：

| 字段         | 类型     | 默认值 | 说明     |
| ------------ | -------- | ------ | -------- |
| `inputAudio` | `string` | `''`   | 音频 URL |

- 上游满足条件：`inputAudio` 非空

### `videoInput`

- 输入：无
- 输出：`video`
- 关键字段：

| 字段                 | 类型     | 默认值 | 说明                                         |
| -------------------- | -------- | ------ | -------------------------------------------- |
| `inputVideo`         | `string` | `''`   | 视频 URL                                     |
| `inputVideoPoster`   | `string` | 无     | 封面 URL                                     |
| `inputVideoDuration` | `number` | 无     | 时长。代码里按毫秒阈值做 motion-control 校验 |

- 上游满足条件：`inputVideo` 非空

## 6.2 文本类任务节点

### `textGenerator`

- 输入：`text` 必填，`image` 可选，`video` 可选，`audio` 可选
- 输出：`text`
- 最大输入连线：无限制
- 运行核心字段：

| 字段                        | 类型           | 默认值                   |
| --------------------------- | -------------- | ------------------------ |
| `inputText`                 | `string`       | `''`                     |
| `selectedModels`            | `string[]`     | `['openai/gpt-4o-mini']` |
| `model_options.attachments` | `Attachment[]` | `[]`                     |

- 运行要求：
  - `status === 'idle'`
  - `selectedModels.length > 0`
  - 满足以下之一：
    - 自己的 `inputText` 有内容
    - 有满足要求的上游输入

### `describeImage`

- 输入：`image` 必填
- 输出：`text`
- 最大输入连线：无限制
- 运行核心字段：

| 字段             | 类型       | 默认值                                 |
| ---------------- | ---------- | -------------------------------------- |
| `inputText`      | `string`   | `''`                                   |
| `selectedModels` | `string[]` | 无固定默认值文档化保证，运行时必须非空 |

- 运行要求：
  - `status === 'idle'`
  - `selectedModels.length > 0`
  - 保守写法：应提供 `image` 上游

### `scriptSplit`

- 输入：`text` 必填
- 输出：`text`
- 最大输入连线：`text` 最多 1 条
- 运行核心字段：

| 字段                        | 类型           | 默认值               |
| --------------------------- | -------------- | -------------------- |
| `inputText`                 | `string`       | `''`                 |
| `selectedModels`            | `string[]`     | `['openai/gpt-5.2']` |
| `model_options.attachments` | `Attachment[]` | `[]`                 |

- 运行要求：
  - `status === 'idle'`
  - 保守写法：应提供 `inputText` 或 1 条 `text` 上游

## 6.3 图像类任务节点

- 输入：`text` 必填
- 输出：`image`
- 最大输入连线：无限制
- 运行核心字段：

| 字段                       | 类型       | 默认值                        |

### `imageToImage`

- 输入：`image` 必填，`text` 必填
- 输出：`image`
- 最大输入连线：无限制
- 运行核心字段：

| 字段               | 类型       | 默认值                               |

| `inputText`        | `string`   | `''`                                 |

- 特殊规则：
  - 如果 `inputText` 本身有内容，前端运行检查会把 `text` 必填 Pin 视为已满足
  - 但保守生成策略仍建议：给它接 `image`，并显式提供 `inputText`

### `imageUpscaler`

- 输入：`image` 必填
- 输出：`image`
- 最大输入连线：`image` 最多 1 条
- 运行核心字段：

| 字段                           | 类型     | 默认值 | 允许值                |
| ------------------------------ | -------- | ------ | --------------------- |
| `model_options.upscale_factor` | `string` | `'2'`  | `'2'` / `'4'` / `'6'` |

### `backgroundEditor`

- 输入：`image` 必填，`text` 可选
- 输出：`image`
- 最大输入连线：`image` 最多 1 条
- 运行核心字段：

| 字段                       | 类型     | 默认值        |
| -------------------------- | -------- | ------------- |
| `inputText`                | `string` | `''`          |
| `model_options.model_mode` | `string` | `'Change BG'` |

- 特殊规则：
  - 如果 `model_mode === 'Change BG'`，则 `inputText` 不能为空
  - 如果 `model_mode === 'Remove BG'`，则可以没有 `inputText`

### `relight`

- 输入：`image` 必填

### `imageAngleControl`

- 输入：`image` 必填
- 输出：`image`
- 最大输入连线：`image` 最多 1 条
- 默认模型：

```json
["fal-ai/qwen-image-edit-2511-multiple-angles"]
```

```json
  "fal-ai/qwen-image-edit-2511-multiple-angles": {
```

- 默认模型：


## 6.6 工具节点

### `assembleNow`

- 输入：`video` 可选，`audio` 可选，`image` 可选
- 输出：`video`
- 可执行：否
- 关键字段：

| 字段              | 类型                   | 默认值 |
| ----------------- | ---------------------- | ------ |
| `assembleAssets`  | `AssembleAsssetItem[]` | `[]`   |
| `assemblePayload` | `string`               | `''`   |

说明：

- 它有输出 Pin，但它被归类为非执行节点
- backend agent 不应该把它当作普通 AI 任务节点来排执行依赖

---

## 6.7 静态节点

### `stickyNodesNode`

- 输入：无
- 输出：无
- 可执行：否
- 作用：便签/视觉标注
- 默认字段：

| 字段              | 类型     | 默认值      |
| ----------------- | -------- | ----------- |
| `inputText`       | `string` | `''`        |
| `backgroundColor` | `string` | `'#DDEEDB'` |
| `stickyMode`      | `string` | `'text'`    |
| `stickyArrow`     | `string` | `'center'`  |
| `stickyRotation`  | `number` | `0`         |
| `imageBase64`     | `string` | `''`        |
| `inputVideo`      | `string` | `''`        |
| `inputAudio`      | `string` | `''`        |

这类字段基本是 UI 数据。除非你的任务就是编辑便签，否则通常可以不理解其业务含义。

### `groupNode`

- 输入：无
- 输出：无
- 可执行：否
- 作用：分组/视觉容器
- 默认字段：

| 字段              | 类型     | 默认值 |
| ----------------- | -------- | ------ |
| `label`           | `string` | 无     |
| `inputHint`       | `string` | 无     |
| `backgroundColor` | `string` | 无     |

## 七、哪些节点“要求已选模型”

当前前端运行校验里，只有下列节点会被硬性检查 `selectedModels.length > 0`：

- `textGenerator`
- `imageMaker`
- `describeImage`
- `videoMaker`
- `imageToImage`
- `klingMotionControl`
- `imageAngleControl`
- `voiceCloner`

这意味着：

- 不是所有任务节点都会在“运行前校验”里强制要求模型非空
- 但从保守建图角度，凡是本身依赖模型的节点，你都应该提供合理的 `selectedModels`

---

## 八、哪些节点“理论上可无上游直接运行”

前端常量中被标记为 `SINGLE_PROMPT_NODE_TYPES` 的有：

- `textToSpeech`
- `describeImage`
- `scriptSplit`
- `textGenerator`
- `imageMaker`
- `musicGenerator`
- `textToVideo`
- `videoLipSync`
- `imageAudioToVideo`

但真正是否能无上游运行，代码还会再检查 `inputText` 是否有有效内容。

因此 backend agent 应采用更保守的规则：

- 只有当节点确实拥有 `inputText` 语义，且 `inputText` 非空时，才把它视为“可独立运行”
- 对本质上依赖媒体输入的节点，不要因为前端常量里被列入 `SINGLE_PROMPT_NODE_TYPES` 就省略媒体输入

---

## 九、节点运行前的保守校验清单

如果 backend agent 需要判断“这个节点现在是否应该被视为可运行”，建议使用下面这套保守规则。

### 9.1 通用规则

1. 节点必须不是非执行节点
2. `data.status` 应为 `idle`
3. 所有必填输入 Pin 都要被满足
4. 所有上游输入节点必须真的有值
5. 如果上游是任务节点，则应假设它必须已有可用产出

### 9.2 文本有效性的定义

对于 `inputText`：

- 不能只看字符串是否非空
- 应该先去掉富文本 HTML 标签
- 去标签后仍然有非空字符，才算有效文本

### 9.3 特殊规则

- `textToSpeech`：必须有已选语音
- `backgroundEditor`：当 `model_mode === 'Change BG'` 时必须有 `inputText`
- `imageToImage`：如果自己 `inputText` 有效，可跳过 text 必填 Pin
- `imageAudioToVideo`：如果自己 `inputText` 有效，可跳过 text 必填 Pin
- `klingMotionControl`：受上游视频时长限制

---

## 十、操作手册

## 10.1 创建一个新节点时，最少需要什么

如果你的任务是“往工作流里新增一个节点”，你至少要提供：

```json
{
  "id": "<唯一ID>",
  "type": "<节点类型>",
  "position": { "x": 0, "y": 0 },
  "selected": false,
  "data": {
    "status": "idle",
    "workflowId": "<当前workflowId>",
    "...": "该节点类型所需的字段"
  }
}
```

### 创建时必须遵循的原则

1. `id` 必须唯一
2. `type` 必须是合法节点类型
3. `data` 里只写这个节点真正需要的字段
4. 不要把别的节点字段抄进来
5. 新节点默认 `status` 应为 `idle`
6. 如果该节点有默认 `selectedModels` / `modelConfigs` / `model_options`，应按当前默认值写入

### 创建时建议保留的展示字段

如果你的输出要直接回流到前端，建议同时保留：

- `data.label`
- `data.description`
- `data.themeColor`
- `data.inputHint`

但这些不属于“执行语义”的核心。

---

## 10.2 修改一个节点时要注意什么

### 可以安全修改

- 本节点自己的业务字段
- `selectedModels`
- `modelConfigs`
- `model_options`
- `inputText`
- 媒体输入字段，例如 `imageBase64` / `inputAudio` / `inputVideo`

### 修改时的约束

1. 不要把节点改成另一个 `type`
2. 不要给节点塞不属于它的字段
3. 改 `selectedModels` 时，要检查 `modelConfigs` 是否仍匹配
4. 改 `model_options` 时，要保持它仍是该节点允许的 schema
5. 改完后，要重新验证这个节点的必填输入是否仍满足

### 什么时候应该一起更新 `modelConfigs`

如果你：

- 替换了 `selectedModels`
- 新增了模型
- 删除了旧模型

则应同步确认：

- 新模型是否需要默认配置
- 旧模型残留配置是否应该删除

---

## 10.3 删除一个节点时，数据结构怎么变化

删除节点时，不是只删 `nodes[]` 里的那一项。

你还必须同步删除所有关联边：

- `edge.source === node.id`
- `edge.target === node.id`

也就是说：

```text
nodes = nodes.filter(node.id != deletedId)
edges = edges.filter(edge.source != deletedId && edge.target != deletedId)
```

删除节点后，下游节点可能会变成“输入不满足，不可运行”。

---

## 10.4 新建连线时要做什么

新增一条边之前，必须做：

1. 检查源节点存在
2. 检查目标节点存在
3. 检查 `sourceHandle` 是否是源节点允许输出的 Pin
4. 检查 `targetHandle` 是否是目标节点允许输入的 Pin
5. 检查两者类型是否兼容
6. 检查目标 Pin 是否超过最大连线数
7. 检查这条边是否会形成环

全部通过后，才能把边 append 到 `edges[]`。

---

## 10.5 删除连线时要做什么

删除边本身很简单：

```text
edges = edges.filter(edge.id != edgeId)
```

但删除前后要意识到：

- 节点 `data` 里通常不存“已连接状态”
- 所以删边时不需要同步写 `isNodeConnected` 之类字段
- 真正的影响是：下游节点的必填输入可能不再满足

---

## 10.6 复制节点时要注意什么

前端当前复制节点的行为不是“只复制节点”，而是：

1. 复制节点本体
2. 复制指向它的入边
3. 复制它指向别人的出边
4. 对复制出来的新边再次检查连接上限
5. 超过限制的边会被丢弃

所以如果 backend agent 实现“复制节点”，不要默认所有边都能原样复制成功。

---

## 十一、对 backend agent 最重要的三条经验

### 11.1 把这套系统当成“有类型的 DAG”

它不是随便连的图，而是：

- 节点有固定类型
- Pin 有固定类型
- 边有方向
- 不能成环
- 某些目标 Pin 还有容量限制

### 11.2 真正影响执行的，只有少数字段

通常最重要的是：

- `node.type`
- `edge.source` / `edge.target`
- `edge.sourceHandle` / `edge.targetHandle`
- `data.inputText`
- `data.imageBase64`
- `data.inputAudio`
- `data.inputVideo`
- `data.selectedModels`
- `data.modelConfigs`
- `data.model_options`
- `data.status`

### 11.3 UI 字段要尽量保留，但不要让它们主导决策

例如：

- `position`
- `label`
- `description`
- `themeColor`

这些字段应该尽量保留，以免前端显示变差；
但 agent 的判断逻辑不应建立在这些字段之上。

---

## 十二、禁止事项

backend agent 不应该做这些事：

1. 创建废弃节点类型
2. 写入垃圾字段
3. 依赖 `data.type` 判断节点类型
4. 创建类型不兼容的边
5. 创建会成环的边
6. 超出 Pin 的最大连线数
7. 给一个节点塞入其他节点的字段
8. 把 `assembleNow`、`stickyNodesNode`、`groupNode` 当成普通可执行节点
9. 只删节点不删关联边
10. 删除边时去维护已经废弃的“连接状态字段”

---

## 十三、如果你要把这份知识库压缩成 agent 心智模型

可以压缩成下面这几句话：

1. 工作流 = `nodes[] + edges[]`
2. 以 `node.type` 为准，不以 `data.type` 为准
3. 每种节点只允许自己的字段，不允许垃圾字段
4. 边必须满足类型兼容、容量限制、无环
5. 节点能否运行取决于：状态、模型/语音选择、必填输入是否满足、上游是否真的有值
6. 删除节点时必须级联删边
7. UI 字段尽量保留，但不要拿它们做执行推理
